# Issue品質評価基準

## 評価アルゴリズム概要

GitHub Issueの品質を0-100点で評価します。満点から減点方式でスコアリングし、低いスコアのIssueを「低品質・曖昧」と判断します。

## 評価項目と配点

### 1. タイトル評価（40点満点）

#### 基準
- **20点減点**: タイトルが10文字未満
- **10点減点**: タイトルが20文字未満
- **15点減点**: 曖昧なタイトルパターンに該当

#### 曖昧なタイトルパターン
- `バグ`, `問題`, `不具合`, `bug`, `problem` （単体）
- `Xバグ`, `Y問題` （5文字以内の接頭辞+一般的な単語）
- `動かない`, `動作しない`, `できない`
- `助けて`, `help`, `sos`
- `要望`, `request`, `feature`
- `X要望` （10文字以内の接頭辞+要望）
- `質問`, `question`

### 2. 本文評価（60点満点）

#### 基準
- **60点減点**: 本文が空
- **30点減点**: 本文が50文字未満
- **15点減点**: 本文が100文字未満

### 3. 情報量ボーナス（加点評価）

以下の情報が含まれている場合、減点を相殺：

| 情報項目 | 加点 | 検出パターン |
|----------|------|-------------|
| 環境情報 | +5点 | `環境`, `environment` |
| 再現手順 | +10点 | `再現`, `reproduce`, `step`, `手順` |
| エラー情報 | +5点 | `エラー`, `error`, `例外`, `exception` |
| 期待動作 | +5点 | `期待`, `expected`, `期望` |
| 実際の動作 | +5点 | `実際`, `actual`, `実際の動作` |
| バージョン情報 | +3点 | `バージョン`, `version` |
| 参考URL | +8点 | `http://`, `https://` |

### 4. その他評価項目

- **10点減点**: ラベルが付与されていない
- **+5点**: コードブロック（```）が含まれている

## 品質判断の目安

### 低品質・曖昧（50点以下）
- タイトルが曖昧で内容が伝わらない
- 本文が極端に短い、または空
- 問題解決に必要な情報が不足

### 要改善（51-70点）
- タイトルは具体的だが本文が不十分
- 一部の情報が欠けている
- 再現性が低い記述

### 品質良好（71-85点）
- タイトルと本文が適切
- 基本的な情報が揃っている
- 修正・改善可能なレベル

### 品質优秀（86-100点）
- 問題解決に十分な情報が含まれている
- 再現手順が明確
- 開発者がすぐに対応可能な状態

## 特殊パターンの処理

### Pull Requestの除外
- Issue番号に`pull_request`キーが含まれる場合、評価対象から除外

### 情報の重複カウント防止
- 同じ情報項目を複数回検出しても、加点は一度のみ
- 情報がある場合は、該当する減点理由から除外

## スコア計算例

### 例1：低品質Issue
- タイトル：「バグ」 (-20点：短い, -15点：曖昧)
- 本文：「動きません」 (-30点：50文字未満)
- ラベル：なし (-10点)

スコア：100 - 20 - 15 - 30 - 10 = 25点

### 例2：品質良好Issue
- タイトル：「ログインボタンがクリックできない」
- 本文：「Chromeでログインページのボタンをクリックしても反応しません」
  + 「環境情報：Chrome 120」 (+5点：環境情報)
  + 「コンソールエラー：Cannot read property 'click'」 (+5点：エラー情報)
- ラベル：bug, high-priority

スコア：100 + 5 + 5 = 100点（上限）
