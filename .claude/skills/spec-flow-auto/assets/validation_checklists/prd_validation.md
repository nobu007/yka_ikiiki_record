# PRD品質チェックリスト

## はじめに

このチェックリストは、高品質なPRD（Product Requirements Document）を作成するための基準です。SpecWorkflowMcpで効果的なSPECを生成するために、PRDが十分な品質を持っていることを確認します。

## 基本構造の確認

### ✅ ドキュメント構造
- [ ] プロジェクト概要セクションが存在する
- [ ] ユーザー像とユースケースセクションが存在する
- [ ] 機能要件セクションが存在する
- [ ] 非機能要件セクションが存在する
- [ ] 受け入れ基準セクションが存在する
- [ ] セクション間の論理的な流れが適切である

### ✅ 文書品質
- [ ] 文章が明確で曖昧さがない
- [ ] 専門用語が適切に定義されている
- [ ] 一貫した用語使用がされている
- [ ] 誤字脱字がない
- [ ] 番号付けが統一されている

## プロジェクト概要の品質

### ✅ 背景・課題
- [ ] 解決すべき課題が具体的に記述されている
- [ ] 現状の問題点が明確に指摘されている
- [ ] ビジネス上のインパクトが説明されている
- [ ] 市場機会や競合状況が考慮されている

### ✅ プロジェクト目的
- [ ] 達成したい目標が具体的に記述されている
- [ ] 成功指標（KPI）が明確に定義されている
- [ ] ビジネス価値が具体的に示されている
- [ ] 定性的・定量的な目標がバランス良く設定されている

### ✅ スコープ定義
- [ ] 含まれる機能が明確にリストアップされている
- [ ] 含まれない機能が明記されている
- [ ] 将来の拡張性が考慮されている
- [ ] フェーズ分け（現在と将来）が明確である

## ユーザー像とユースケースの品質

### ✅ ユーザー像（Persona）
- [ ] 主要ユーザー像が定義されている（最低2-3種類）
- [ ] 各ユーザーの背景情報が具体的である
- [ ] スキルレベルや技術的な知識レベルが記述されている
- [ ] 課題とニーズが明確に記述されている

### ✅ ユースケース
- [ ] 主要ユースケースが網羅的に定義されている
- [ ] 各ユースケースが「誰が・何を・なぜ・期待結果」の形式で記述されている
- [ ] ユースケース間の関係性が考慮されている
- [ ] 主要ユースケースと副次的なユースケースの区別が明確である

## 機能要件の品質

### ✅ 機能要件の網羅性
- [ ] 全ての主要機能が記述されている
- [ ] 各機能にFR-XXX形式のIDが付与されている
- [ ] 機能の説明が具体的で詳細である
- [ ] 入力・出力・ビジネスルールが明記されている

### ✅ UI/UX要件
- [ ] 主要画面の構成が記述されている
- [ ] ユーザー操作フローが明確である
- [ ] データ表示要件が定義されている
- [ ] UXに関する考慮事項が記述されている

### ✅ データ要件
- [ ] 扱うデータの種類が明確である
- [ ] データ形式や構造が定義されている
- [ ] データ量と成長予測が考慮されている
- [ ] データ保持期間に関する要件がある

## 非機能要件の品質

### ✅ 性能要件
- [ ] レスポンスタイム要件が数値で定義されている
- [ ] 同時接続ユーザー数が明記されている
- [ ] データ処理能力要件がある
- [ ] スループット要件が定義されている

### ✅ セキュリティ要件
- [ ] 認証・認方式が具体的に定義されている
- [ ] データ保護要件がある
- [ ] アクセス制御要件が明確である
- [ ] 監査・ログ要件が記述されている

### ✅ 可用性要件
- [ ] システム稼働率が数値目標で定義されている
- [ ] メンテナンス時間に関する要件がある
- [ ] バックアップ・リカバリ要件がある
- [ ] 災害対策が考慮されている

### ✅ 互換性要件
- [ ] 対応ブラウザが明記されている
- [ ] 対応OSが定義されている
- [ ] APIバージョン互換性が考慮されている
- [ ] モバイル対応要件がある（必要な場合）

## 技術要件の品質

### ✅ 技術スタック
- [ ] 使用技術が具体的にリストアップされている
- [ ] バージョン指定が適切である
- [ ] 技術選定の理由が説明されている
- [ ] 既存システムとの整合性が考慮されている

### ✅ 外部連携
- [ ] 外部APIやサービスの連携要件が明確である
- [ ] 認証方式やデータ形式が定義されている
- [ ] 制約事項（レートリミット等）が記述されている
- [ ] フォールバック戦略が考慮されている

## 制約事項と前提条件

### ✅ 制約事項
- [ ] 技術的制約が明確に記述されている
- [ ] 事業的制約（期間・予算・人員）が定義されている
- [ ] 法規制に関する制約が考慮されている
- [ ] 既存システムに関する制約がある

### ✅ 前提条件
- [ ] 実施前提となる条件が記述されている
- [ ] 依存関係が明確である
- [ ] 必要なリソースが定義されている
- [ ] リスク要因が考慮されている

## 受け入れ基準の品質

### ✅ 受け入れ基準
- [ ] 機能的受け入れ基準が具体的である
- [ ] 非機能的受け入れ基準が測定可能である
- [ ] 品質基準（テスト、レビュー等）が定義されている
- [ ] 基準達成の確認方法が明確である

### ✅ 成功指標
- [ ] ビジネス指標が定義されている
- [ ] 技術指標が数値目標で設定されている
- [ ] プロジェクト指標がある
- [ ] 測定方法と測定頻度が考慮されている

## リスク評価の品質

### ✅ リスク分析
- [ ] 技術的リスクが識別されている
- [ ] 事業的リスクが考慮されている
- [ ] 各リスクの発生確率と影響度が評価されている
- [ ] 具体的な対策が記述されている

## 品質評価基準

### 🟢 良好（80%以上）
- ほぼ全ての項目を満たしている
- SpecWorkflowMcpでのSPEC生成が期待できる
- チームでのレビューも推奨

### 🟡 要改善（60-79%）
- 多くの項目を満たしているが、一部不足がある
- 追加情報の記述で品質向上が可能
- SPEC生成前に改善を推奨

### 🔴 再作成要（59%以下）
- 多くの必須項目が不足している
- SPEC生成の質が低くなる可能性がある
- PRDの全面的な見直しが必要

## 次のアクション

### ✅ 品質向上のヒント

1. **具体化**: 曖昧な表現を具体的・定量的に修正
2. **網羅性**: 漏れている視点や要件を追加
3. **測定可能性**: 受け入れ基準や成功指標を数値化
4. **優先順位**: Must-have, Should-haveの明確化
5. **検証**: ステークホルダーによるレビュー実施

### ✅ SPEC生成準備完了の条件

- チェックリスト評価が80%以上
- 主要ステークホルダーによる承認
- 要件変更管理プロセスの確立
- SPEC生成スケジュールの設定
