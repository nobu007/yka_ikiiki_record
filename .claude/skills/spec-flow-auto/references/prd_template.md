# PRD (Product Requirements Document) Template

## はじめに

このテンプレートは、高品質なPRDを作成するための標準形式です。SpecWorkflowMcpとの連携を前提に設計されています。

## PRDの基本構造

### 1. プロジェクト概要 (Project Overview)

```markdown
# [プロジェクト名] PRD

## 1. プロジェクト概要

### 1.1 背景・課題
- 現状の問題点
- 解決すべきビジネス課題
- 市場機会

### 1.2 プロジェクトの目的
- 達成したい目標
- 成功指標（KPI）
- ビジネスインパクト

### 1.3 スコープ
- 含まれる機能
- 含まれない機能
- 将来の拡張性
```

### 2. ユーザー像とユースケース (User Personas & Use Cases)

```markdown
## 2. ユーザー像とユースケース

### 2.1 主要ユーザー像
**Persona 1: [役割名]**
- 年齢、職業、背景
- スキルレベル
- 課題とニーズ

### 2.2 主要ユースケース
**UC-001: [ユースケース名]**
- 誰が：[ユーザー像]
- 何を：[具体的なアクション]
- なぜ：[目的・ゴール]
- 期待結果：[具体的な成果]
```

### 3. 機能要件 (Functional Requirements)

```markdown
## 3. 機能要件

### 3.1 コア機能
**FR-001: [機能名]**
- 説明：機能の詳細説明
- 入力：必要なデータ・情報
- 出力：期待される結果
- ビジネスルール：制約条件

### 3.2 ユーザーインターフェース要件
**UI-001: [画面名]**
- 画面構成要素
- ユーザー操作フロー
- データ表示要件
```

### 4. 非機能要件 (Non-Functional Requirements)

```markdown
## 4. 非機能要件

### 4.1 性能要件
- レスポンスタイム：2秒以内
- 同時接続ユーザー数：1000人以上
- データ処理量：1GB/分

### 4.2 セキュリティ要件
- 認証方式：JWTベース
- データ暗号化：AES-256
- 監査ログ：全操作記録

### 4.3 可用性要件
- システム稼働率：99.9%以上
- メンテナンス時間：月4時間以内
- データバックアップ：日次自動バックアップ

### 4.4 互換性要件
- ブラウザ対応：Chrome, Firefox, Safari最新版
- OS対応：Windows 10+, macOS 10.15+, Ubuntu 18.04+
- モバイル対応：iOS 14+, Android 10+
```

### 5. 技術要件 (Technical Requirements)

```markdown
## 5. 技術要件

### 5.1 技術スタック
- フロントエンド：React 18+, TypeScript
- バックエンド：FastAPI, Python 3.11+
- データベース：PostgreSQL 15+
- インフラ：Docker, AWS/Azure

### 5.2 外部連携
- 決済ゲートウェイ：Stripe API
- 認証プロバイダ：OAuth 2.0
- 通知サービス：Email, Push通知

### 5.3 データ要件
- データ形式：JSON, CSV
- データ量：初期1TB、年間20%増加
- 保持期間：7年間
```

### 6. 制約事項と前提条件 (Constraints & Assumptions)

```markdown
## 6. 制約事項と前提条件

### 6.1 技術的制約
- 使用可能な技術：既定の技術スタック内
- 外部API制限：レートリミット1000リクエスト/時間
- データ容量：ストレージ容量上限

### 6.2 事業的制約
- 開発期間：最大6ヶ月
- 予算上限：既定予算内
- 人員：最大5名

### 6.3 前提条件
- 既存システムとの連携必須
- 社内承認プロセスの遵守
- 法規制の遵守
```

### 7. 受け入れ基準 (Acceptance Criteria)

```markdown
## 7. 受け入れ基準

### 7.1 機能的受け入れ基準
- [ ] 全ての機能要件が仕様通りに動作する
- [ ] ユーザーが主要タスクを完了できる
- [ ] エラー処理が適切に行われる

### 7.2 非機能的受け入れ基準
- [ ] 性能要件を満たす（レスポンス2秒以内）
- [ ] セキュリティテストに合格する
- [ ] 負荷テストで目標値を達成する

### 7.3 品質基準
- [ ] コードカバレッジ80%以上
- [ ] セキュリティ脆弱性ゼロ
- [ ] ユーザー受け入れテスト合格率90%以上
```

### 8. リスク評価 (Risk Assessment)

```markdown
## 8. リスク評価

### 8.1 技術的リスク
**高リスク：データベースパフォーマンス**
- 発生確率：中
- 影響度：大
- 対策：早期負荷テスト実施

### 8.2 事業的リスク
**中リスク：要求仕様の変更**
- 発生確率：高
- 影響度：中
- 対策：アジャイル開発手法の採用
```

### 9. 成功指標 (Success Metrics)

```markdown
## 9. 成功指標

### 9.1 ビジネス指標
- ユーザー満足度：4.5/5.0以上
- 業務効率化：30%以上改善
- コスト削減：20%以上達成

### 9.2 技術指標
- システム稼働率：99.9%以上
- 平均レスポンスタイム：500ms以下
- バグ解決時間：24時間以内

### 9.3 プロジェクト指標
- 納期遵守率：100%
- 予算遵守率：95%以上
- 品質スコア：85点以上
```

## SpecWorkflowMcp連携のヒント

### 高品質なPRDの条件

1. **具体的であること**
   - 曖昧な表現を避ける
   - 数値目標を明示する
   - 具体的なユーザー行動を記述する

2. **網羅性があること**
   - 全ての機能要件を網羅
   - 非機能要件を十分に記述
   - 制約事項を明確にする

3. **優先順位が明確であること**
   - Must-have, Should-have, Could-haveの区分
   - 機能間の依存関係の明確化
   - 実装順序の考慮

4. **検証可能であること**
   - 具体的な受け入れ基準
   - 測定可能な成功指標
   - テスト可能な要件定義

### SPEC生成の自動化

このテンプレートに従ってPRDを作成することで、SpecWorkflowMcpが以下を自動生成できます：

- **requirements.md**: 機能要件・非機能要件の構造化
- **design.md**: 技術設計・アーキテクチャの具体化
- **tasks.md**: 実装タスクの詳細分解

### 次のステップ

PRDが完成したら、以下のコマンドでSPEC生成を開始できます：

```
"README.mdのPRDから仕様書を生成してください"
```

これにより、SDD（仕様駆動開発）の自動プロセスが開始されます。
