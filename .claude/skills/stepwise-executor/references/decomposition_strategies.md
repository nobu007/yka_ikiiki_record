# 目標分解の戦略

このドキュメントは、作業目標を適切な中間目標に分解するための戦略とベストプラクティスを説明します。

## 基本原則

### 1. SMART原則

各中間目標は以下の基準を満たすべきです:

- **Specific（具体的）**: 何をするかが明確
- **Measurable（測定可能）**: 完了したかどうかが判断できる
- **Achievable（達成可能）**: 実現可能な範囲
- **Relevant（関連性）**: 最終目標に貢献する
- **Time-bound（期限付き）**: 推定作業量が示されている

### 2. 適切な粒度

中間目標は大きすぎても小さすぎても良くありません:

- **大きすぎる**: 1つのステップに複数日かかる場合は分割を検討
- **小さすぎる**: 15分未満のタスクは他のステップと統合を検討
- **推奨**: 1ステップあたり30分〜4時間程度

### 3. 依存関係の明確化

ステップ間の依存関係を明確にします:

- **順次依存**: ステップAが完了しないとステップBが開始できない
- **並列実行**: 独立したステップは並列実行可能
- **部分依存**: ステップAの一部の成果物がステップBに必要

## 分解アプローチ

### トップダウンアプローチ

最終目標から逆算して分解する方法:

1. 最終目標を定義
2. 大まかなフェーズに分割（3〜7フェーズ程度）
3. 各フェーズを具体的なステップに分割
4. 各ステップの依存関係を確認

**適用場面**:
- 明確な最終成果物がある場合
- プロジェクト全体の構造が見えている場合
- 計画的なアプローチが求められる場合

**例**: Webアプリケーション開発
```
最終目標: ユーザー認証機能付きのTodoアプリ
↓
フェーズ1: 要件定義と設計
フェーズ2: バックエンド実装
フェーズ3: フロントエンド実装
フェーズ4: テストとデプロイ
↓
フェーズ2の詳細:
  - データベーススキーマ設計
  - ユーザー認証API実装
  - Todo CRUD API実装
  - API統合テスト
```

### ボトムアップアプローチ

必要なタスクをリストアップして整理する方法:

1. 思いつく限りのタスクをリストアップ
2. 類似タスクをグループ化
3. グループごとに順序を整理
4. 依存関係を追加

**適用場面**:
- 全体像が不明確な探索的プロジェクト
- 既存システムの改修・リファクタリング
- 複雑な問題の解決

**例**: レガシーコードのリファクタリング
```
タスクリスト:
- テストカバレッジの確認
- 重複コードの特定
- 依存関係の分析
- データベース設計の見直し
- ...
↓
グループ化:
グループ1（現状分析）:
  - テストカバレッジの確認
  - 依存関係の分析
  - パフォーマンスボトルネックの特定
グループ2（設計改善）:
  - データベース設計の見直し
  - アーキテクチャ設計
グループ3（実装）:
  - 重複コードの統合
  - モジュール分離
  ...
```

### マイルストーンアプローチ

重要なマイルストーンを設定して、その間を埋める方法:

1. プロジェクトの主要マイルストーンを特定
2. マイルストーン間の橋渡しとなるステップを追加
3. 各ステップを具体化

**適用場面**:
- 長期プロジェクト
- 段階的なリリースが必要な場合
- チーム間の調整が必要な場合

**例**: 新機能のリリース
```
マイルストーン:
M1: プロトタイプ完成（2週間後）
M2: ベータ版リリース（1ヶ月後）
M3: 本番リリース（6週間後）
↓
M1までのステップ:
  - 要件定義
  - UI/UXデザイン
  - コアロジック実装
  - 基本UIの実装
M1→M2のステップ:
  - ユーザーフィードバック収集
  - バグ修正
  - パフォーマンス最適化
  ...
```

## 分解時の注意点

### 1. 過度な細分化を避ける

- タスクが多すぎると管理コストが増大
- 目安: 1プロジェクトあたり3〜15ステップ程度
- 15ステップを超える場合は、プロジェクトの分割を検討

### 2. 実行可能性を優先

- 理想的な分解よりも、実際に実行できる分解を優先
- 不確実性が高い部分は「調査」ステップを設ける
- 必要に応じて後から細分化できるよう柔軟に設計

### 3. バッファを含める

- 想定外の問題に対処するバッファステップを含める
- 例: 「統合テストと問題修正」「ドキュメント整備」

### 4. レビューポイントを設定

- 重要な分岐点では判断・確認のステップを設ける
- 例: 「設計レビュー」「パフォーマンス評価」「セキュリティチェック」

## 分解の品質チェックリスト

優れた目標分解は以下の特徴を持ちます:

- [ ] 各ステップが具体的で実行可能
- [ ] ステップの完了条件が明確
- [ ] 依存関係が適切に定義されている
- [ ] 推定作業量が現実的
- [ ] 全体として最終目標を達成できる
- [ ] 並列実行可能なステップが特定されている
- [ ] リスクの高いステップが早期に配置されている
- [ ] 定期的なレビューポイントが含まれている

## 参考: 分解パターン

### ウォーターフォール型

```
要件定義 → 設計 → 実装 → テスト → デプロイ
```

**適用**: 要件が明確で変更が少ないプロジェクト

### イテレーティブ型

```
サイクル1: 設計 → 実装 → テスト
サイクル2: 設計 → 実装 → テスト
サイクル3: 設計 → 実装 → テスト
```

**適用**: 段階的に機能を追加するプロジェクト

### スパイク型

```
技術調査 → プロトタイプ → 評価 → 本実装
```

**適用**: 技術的不確実性が高いプロジェクト
