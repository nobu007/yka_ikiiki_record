## 直近の優先成果（次に終わらせるべきこと）

### P1: E2Eテスト基盤の確立（次に着手すべき最優先タスク）

**なぜE2Eテストが先か**: データ永続化を追加する前に、現在のMirageベースの機能が正しく動作することをE2Eで保証する必要がある。さもないと、データ永続化追加時に「元から壊れていた」のか「永続化追加で壊れた」のかが判別できない。

**完了条件**:
1. Playwrightのセットアップと初期設定
2. 主要ユーザーフローのE2Eテストシナリオ定義
3. 少なくとも以下のフローをカバー:
   - ランディングページ → ダッシュボード遷移
   - データ生成フロー（生徒数・日数・感情種類の設定）
   - 統計データの表示確認
   - エモーションチャートの描画
4. E2EテストがCIで自動実行される設定

**期待される成果物**:
- `playwright.config.ts` の設定
- `e2e/` ディレクトリ以下のテストファイル（少なくとも3-5シナリオ）
- E2Eテストの実行スクリプト（`npm run test:e2e`）

### P2: データ永続化レイヤーの設計と実装

**完了条件**: Mirageモックから実際のデータベースへの移行パスを確立

**決定が必要な事項**:
1. **データベース選定**:
   - 開発環境: SQLite（軽量、セットアップ簡単）
   - 本番環境: PostgreSQL（推奨）またはSupabase
2. **ORM選定**: Prisma（型安全、マイグレーション管理が容易）
3. **移行戦略**: Mirage → Prismaの段階的移行

**実装タスク**:
1. Prismaスキーマの定義（既存のデータモデルを基に）
2. Prisma Clientのセットアップ
3. Repositoryパターンの実装:
   - `IStatsRepository` インターフェース（Domain層）
   - `PrismaStatsRepository` 実装（Infrastructure層）
   - Mirage実装からの置き換え
4. データベースマイグレーションの設定
5. シードデータの実装（初期データ生成）

**注意点**: Clean Architectureを維持すること。Domain層はPrismaに依存してはいけない。Interfaceを通じてのみ依存すること。

### P3: 認証・認可機能の追加

**完了条件**: ユーザー認証と基本的な認可が実装されている

**実装タスク**:
1. 認証プロバイダーの選定（NextAuth.js推奨）
2. ユーザーモデルの実装
3. ログイン・ログアウトフローの実装
4. APIルートの保護（認証済みユーザーのみアクセス可能）
5. E2Eテストでの認証フローの検証

## 技術方針

### データ永続化のアーキテクチャ
```
Domain層（インターフェースのみ）:
  - IStatsRepository: findById, save, delete, findAll

Infrastructure層（実装）:
  - MirageStatsRepository（現在、開発用）
  - PrismaStatsRepository（将来、本番用）

Application層:
  - StatsService（Domain層のリポジトリを使用）
```

**重要**: Domain層はPrismaを知らない。Interfaceのみを定義する。

### E2Eテスト方針
- Playwrightを使用
- ページオブジェクトモデル（POM）パターンを採用
- テストは独立して実行可能にする
- CI/CDパイプラインに組み込む

### データベースマイグレーション方針
- Prisma Migrateを使用
- 開発と本番で異なるデータベースをサポート
- マイグレーションファイルはバージョン管理する
